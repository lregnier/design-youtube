## Context

Both bounded contexts follow hexagonal architecture with explicit port interfaces. The application layer (`internal/application/`) has zero infrastructure imports — every dependency is injected via an interface. This makes mockery-based unit testing straightforward: generate mocks for each interface, inject them into use cases, assert behaviour without any network or disk I/O.

## Goals / Non-Goals

**Goals:**
- Full coverage of happy path and meaningful error paths for every application-layer use case
- AAA pattern enforced with inline comments in every test function
- Mocks generated by mockery v2 from the existing interface definitions
- Tests run with `go test ./internal/application/...` in each module — no external dependencies required

**Non-Goals:**
- Integration tests (require LocalStack / real AWS)
- Adapter tests (DynamoDB, S3, ffmpeg, Redis, SQS adapters)
- Domain tests (pure structs with no logic worth mocking)
- 100% line coverage — meaningful scenarios only, not coverage theatre

## Decisions

### 1. mockery v2 with `inpackage: false` and a top-level `mocks/` directory

Each module gets a `.mockery.yaml` that points mockery at the relevant interfaces and outputs mocks to `internal/mocks/`. This keeps generated code out of the interface packages and avoids circular imports.

```yaml
# backend/api/.mockery.yaml
quiet: true
disable-version-string: true
with-expecter: true
packages:
  github.com/lregnier/design-youtube/api/internal/domain/video:
    interfaces:
      VideoRepository:
  github.com/lregnier/design-youtube/api/internal/ports:
    interfaces:
      ObjectStore:
      Cache:
output: internal/mocks
```

`with-expecter: true` generates type-safe `EXPECT()` helpers — cleaner than raw `On("Method", ...)` strings.

### 2. Test files co-located with use cases

`init_test.go` lives next to `init.go` in `internal/application/upload/`. Same package (`package upload`) — tests can access unexported helpers if needed. No separate `test/` directory.

### 3. AAA pattern with explicit comments

Every test function uses the three-section structure:

```go
func TestInitUpload_Execute_ValidRequest(t *testing.T) {
    // Arrange
    repo := mocks.NewMockVideoRepository(t)
    store := mocks.NewMockObjectStore(t)
    repo.EXPECT().Save(mock.Anything, mock.Anything).Return(nil)
    store.EXPECT().CreateMultipartUpload(mock.Anything, mock.Anything).Return(...)
    store.EXPECT().PresignUploadPart(mock.Anything, ...).Return(...)
    uc := NewInitUpload(repo, store, "my-bucket")

    // Act
    result, err := uc.Execute(ctx, InitUploadCommand{...})

    // Assert
    assert.NoError(t, err)
    assert.Equal(t, "part-1-url", result.PresignedURL)
}
```

### 4. Test naming convention: `Test<UseCase>_<Method>_<Scenario>`

Examples:
- `TestInitUpload_Execute_ValidRequest`
- `TestInitUpload_Execute_FileTooLarge`
- `TestInitUpload_Execute_ResumeExistingUpload`
- `TestGetVideo_Execute_CacheHit`
- `TestGetVideo_Execute_CacheMissPopulatesCache`
- `TestProcessVideo_Execute_TranscodeFailureEmitsVideoFailed`

### 5. Scenarios to cover per use case

**InitUpload**: valid new upload, file exceeds 100MB, resume existing uploading video, repo.Save error.

**ConfirmChunk**: chunk confirmed with next part remaining, final chunk confirmed (done=true), video not found.

**CompleteUpload**: successful completion updates status to processing, video not found, S3 CompleteMultipartUpload error.

**GetVideo**: cache hit returns without DynamoDB call, cache miss queries repo and populates cache, video not found returns nil.

**ListVideos**: returns ready videos, empty list when none.

**ApplyProcessingResult**: VideoProcessed marks ready, VideoProcessed idempotent on already-ready video, VideoFailed marks failed, video not found returns error.

**ProcessVideo (worker)**: successful full pipeline emits VideoProcessed, download failure emits VideoFailed, transcode failure emits VideoFailed, thumbnail failure is non-fatal (still emits VideoProcessed).

## Risks / Trade-offs

- **Generated mocks need regenerating when interfaces change** → run `mockery` after any interface modification. Add to contributing notes.
- **`with-expecter` generates additional boilerplate** → cleaner call sites outweigh the generated code verbosity.
- **Test isolation via `t` cleanup** → mockery v2 mocks created with `NewMockX(t)` auto-assert expectations on `t.Cleanup` — no manual `AssertExpectations` needed.
