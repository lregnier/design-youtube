## Context

The worker bounded context was built before the api's architectural patterns were finalised. As a result it has several structural differences:

| Concern | api (current) | worker (current) |
|---|---|---|
| Infra dirs | `infrastructure/in/`, `infrastructure/out/` | `adapters/inbound/`, `adapters/outbound/` |
| Port interfaces | co-located in `application/` | separate `ports/` package |
| Domain events | `domain/`, no JSON tags | `internal/event/`, with JSON tags |
| Mocks | `gen/mocks/` | `internal/mocks/` |
| Config | `cmd/api/config.go` (package main) | `internal/config/` |
| Use case | interface + unexported struct | exported concrete struct |

## Goals / Non-Goals

**Goals:**
- Make `internal/` contain only three layers: `domain/`, `application/`, `infrastructure/`
- Port interfaces live in `application/` alongside the use case that consumes them
- Domain events live in `domain/processing/`, tag-free; JSON serialization stays in the `sqspublisher` adapter
- Mocks generated by mockery live in `gen/mocks/`
- Config lives in `cmd/worker/config.go` as `package main`
- `ProcessVideo` becomes an interface; `sqsjobs` consumer depends on the interface

**Non-Goals:**
- Changing any business logic, processing algorithms, or SQS polling behaviour
- Changing the wire format of SQS messages

## Decisions

### `infrastructure/in/` and `infrastructure/out/` over `adapters/inbound/` and `adapters/outbound/`

Consistent naming with the api makes both contexts immediately readable as the same architecture. `infrastructure/` is the conventional DDD term; `in/out` is compact and unambiguous.

### Port interfaces merged into `application/`

The `ports/` package is an extra layer with a single file. Merging its interfaces into `application/` alongside the use case that consumes them removes the indirection with no loss of separation — the interfaces are still defined before any concrete adapter sees them.

### Events renamed to match api convention

`VideoProcessed` → `VideoProcessingSucceededEvent`, `VideoFailed` → `VideoProcessingFailedEvent`. Consistent naming across both contexts eliminates the mental translation overhead when reading cross-context code.

### `sqspublisher` wire structs

The publisher currently uses `event.VideoProcessed` / `event.VideoFailed` which carry JSON tags. After the move, it will use unexported local wire structs (same pattern as `sqssubscriber` in the api). Domain event types carry no JSON tags.

### `ProcessVideo` as interface

The `sqsjobs` consumer currently accepts `application.ProcessVideo` (concrete struct). Making it an interface allows the consumer to be tested with a mock use case — the same pattern the api uses for all application services. The unexported `processVideo` struct is the implementation; `NewProcessVideo` returns the interface.

### mockery for mocks

The worker currently has hand-written mocks in `internal/mocks/`. Switching to mockery-generated mocks (same as api) ensures they stay in sync with interface changes and removes manual maintenance.

## Risks / Trade-offs

[Broad import churn] → Every file in `cmd/worker/main.go`, `sqsjobs/consumer.go`, and `sqspublisher/publisher.go` needs import path updates. All mechanical — no logic changes.

[Mock regeneration] → Existing hand-written mocks will be deleted and replaced by generated ones. The generated mocks use the testify expecter pattern (same as api tests) which may require minor test updates to use the new mock API.

[ProcessVideo interface] → Consumer test currently tests against the concrete struct. After the change it tests against the interface via a generated mock — a net improvement in isolation.
